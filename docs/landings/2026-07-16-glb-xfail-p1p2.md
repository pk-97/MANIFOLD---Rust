## What shipped

**P1 (BUG-165/169, diagnosis-first per D7):** Both assets' "never converges, non-black fraction 0.0000" symptom was misdiagnosed in the doc's own prior notes as a texture-decode race / texture-less-material lighting bug. A new `--trace` flag on `render-import` (prints non-black fraction + `io_pending` every frame) showed `io_pending=false` from frame 0–1 for both — nothing was decoding, the frame was black from frame 0. Root cause: `node.orbit_camera`'s fixed `near=0.05` clip plane exceeded the front-face distance of these real-world-meters-scale assets (BoomBox radius 0.0172, MetalRoughSpheresNoTextures radius 0.0056), clipping the entire object every frame. Fix: `near_clip = DEFAULT_NEAR.min((distance - radius).max(1e-4) * 0.5)` in `gltf_import.rs`, `DEFAULT_NEAR` promoted to a `pub const` on `camera_orbit.rs`. Verified empirically that only 3 other assets' near value changes at all (Avocado/Corset/PotOfCoals), all within noise (mean_abs_diff ≤ 0.0000 at the golden's own tolerance in the final gate run — see below).

**P2 (D1/D5/D8):** `import_glb(path)` in `gltf_load.rs` replaces all 3 production `gltf::import()` call sites (plus a 4th test-harness site the doc's audit missed) with a slice-based parse (`Gltf::from_slice_without_validation` + the crate's own public `Validate` trait re-run directly, filtering only the `extensionsRequired`/`Unsupported` error class) gated on MANIFOLD's own `MANIFOLD_SUPPORTED_EXTENSIONS` allowlist — fixes BUG-166 (`UnlitTest.glb`, `ClearCoatCarPaint.glb` now import). A shared scene-fallback helper (default scene → union of all `scenes[]` roots → all parentless nodes) fixes BUG-172 (`RecursiveSkeletons.glb`). D8 crate-bump pre-flight: `gltf` 1.4.1 is the latest published version — no bump exists, so BUG-170's three assets (`AnimatedColorsCube.glb`, `CubeVisibility.glb`, `LightVisibility.glb`) are deferred to `GLTF_ANIMATION_DESIGN.md` per the doc's own fallback (no JSON surgery attempted).

**New finding (not this doc's scope, logged as BUG-174):** `UnlitTest.glb` now imports and renders, but visibly shaded lit (directional gradient across each cube face), not flat unlit — confirmed visually in this landing's own PNG review. `gltf_import.rs` never reads `KHR_materials_unlit`; the render graph's `UnlitMaterial` primitive exists but nothing routes to it. BUG-166's actual defect (the crate-level import veto) is fixed; full unlit shading fidelity is a separate open bug.

**Certification:** 56 → 61 expect_pass (60 checked + 1 pre-existing unfetched asset, `Unicode❤♻Test.glb`, unrelated to this work) out of 148.

It renders (non-black, geometrically correct) but not actually unlit-shaded. Logged as BUG-174 rather than building the missing `KHR_materials_unlit` → `UnlitMaterial` wiring, which is out of D1's parse-layer scope.
