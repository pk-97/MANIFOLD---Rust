## Deviations from the design doc's literal acceptance-flow wording

The design's §"Acceptance demo" text reads: *"scrub Roughness via the real card slider in the panel, open its drawer inline, toggle a driver, then undo ONCE and assert the pre-scrub value returns."* Built as specified through "toggle a driver"; the final clause doesn't hold literally once you look at what `PanelAction::DriverToggle` actually does — it dispatches its own real, separate undo-tracked command (`AddDriverCommand`/`ToggleDriverEnabledCommand`, both routed through `ContentCommand::Execute`), so ONE undo after toggling a driver undoes the *driver*, not the scrub. The flow now proves the three underlying capabilities honestly (drag is one commit; the driver button's click dispatches correctly; undo is available) without asserting a physically incorrect invariant. The rigorous "one drag = one undo unit, and that unit's undo restores the pre-drag value" claim is proven where it always has been for this design (C-P1a onward): at the Rust dispatch/UI level (`roughness_card_row_sweeps_full_range_and_value_box_opens_typein` proves the panel emits exactly one `ParamCommit` per gesture; `SetGraphNodeParamCommand`'s own unit tests in `manifold-editing` prove that command's undo restores the captured previous value). Escalated in spirit (documented here), not asked live mid-session — same convention C-P1b's own `build_mod_button`-retention deviation used.

## Escalations — expected, found, and one open question

**Expected and found:** the headless `--script` harness's known BUG-234/BUG-239 gap (driver/envelope modulation and post-drag displayed values are unobservable in the same run that dispatches them) recurred for Light's Intensity row, Modifier's Angle row, and Roughness's driver-drawer — same family, not a new mechanism. Also newly observed and folded into BUG-239's entry: the harness's `Key{Z, command:true}` undo handler reports `undo_count=0` before and after every press in these same runs, even immediately after a real `ParamCommit`/`DriverToggle` dispatch — the harness's undo-stack state is *also* unobservable headlessly, not just its display state. This means the drawer-opens-inline half of the acceptance gate is proven at L2 (a direct Rust render test, `roughness_driver_active_renders_the_drawer_inline`, feeding `RowModulation.driver_active = true` and asserting the drawer's `"Straight"`/`"Dotted"`/`"Triplet"` feel-row text renders) rather than L3 (through the flow script) — consistent with how C-P1a treated the identical class of gap.

**One open question for Peter:** `PanelAction::SceneSetupEnumClicked` (the old P4/D9 dropdown-on-click mechanism) now has zero producers anywhere in the codebase — Modifier's Axis rows were its last one, and C-P1d moved them onto the card's own `value_labels` path. The variant, and its two dispatch-side consumers (`ui_bridge/mod.rs`, `ui_root.rs`), still compile cleanly (no dead-code lint fires for a pattern-matched enum variant), so nothing is broken — but it's now dead weight. Left in place this session (removing it means touching two more files outside `scene_setup_panel.rs`'s own blast radius, and wasn't in the session's explicit gate list); flagging rather than silently leaving it, per the "no silent debt" standard. Fine to clean up in any future small session that touches the panel's dispatch surface.

## Click-script for Peter (≤2 minutes)

1. **Instant selection.** Open Scene Setup (the "Scene" toggle). Click any outliner row (Camera / World / a light / an object) — the Properties body below switches to that item's own rows immediately, no lag.
2. **Card-row scrub, one-undo-per-gesture, every family.** Pick any numeric row — try Fog Density (under World), an object's Position/Rotation/Scale/Roughness, a Light's Intensity, or a Modifier's Angle/Amount param (add one via "+ Add Modifier" if none present) — and drag its slider track end to end. One `Cmd+Z` afterward returns it exactly to where the drag started, regardless of which family the row belongs to.
3. **Inline drawers.** Click the small "T" (driver) icon on any of those same rows — a beat-division / waveform config drawer opens directly in the dock, no separate window. Click "∿" (envelope) or "A" (audio) for their own drawers, same place.
4. **Exposure mod buttons.** The "∼" glyph past a row's D/E/A icons adds that param to the generator card on the right (Macros panel) — click it once on Roughness or a Light's Intensity and check it shows up over there, named `<ObjectName/LightName> · <Param>`.
5. **Eye slots.** Object rows in the outliner carry a visibility eye toggle; Camera/World/Light rows show an intentionally empty slot (no dead-looking glyph) — this is by design (D1 amendment), not a bug.
6. **Viewport + gizmos.** Unaffected by this session — no changes to `manifold-renderer`'s render or gizmo paths; scene rows write through the exact same `SetGraphNodeParamCommand` path they always did, just via new widgets.
