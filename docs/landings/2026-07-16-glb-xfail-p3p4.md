## What shipped

**P3 (D2 spec-gloss conversion + D4 default material, BUG-167/171):** `KHR_materials_pbrSpecularGlossiness` converts to metal-rough at `gltf_load.rs` parse time — diffuseFactor/diffuseTexture → base color, `roughness = 1 - glossinessFactor`, `specularFactor` (RGB mean) → the existing scalar `specular_factor` F0 slot, gloss-channel repack via a new `gltf_texture_source` `mode=gloss_to_roughness` blit option (no shader fork — `render_scene.wgsl` stays spec-gloss-unaware, per D2's explicit rejection of a native spec-gloss BRDF). `SpecGlossVsMetalRough.glb` imports and both halves are now legible and lit, but not pixel-alike — the specularGlossinessTexture's RGB specular tint stays correctly Deferred (§8), and this asset is exactly the trigger that section named for reviving it. D4: a synthetic default-material entry (glTF spec default: white/metallic 1.0/roughness 1.0, sentinel `material_index = u32::MAX`) makes `BoxVertexColors.glb` import successfully for the first time (previously errored "no materials with geometry"). The escalation that mattered: COLOR_0 (per-vertex color) turned out to require adding a field to `MeshVertex`, the shared 48-byte GPU vertex format used by ~30 mesh primitives across the render graph — a structural GPU-ABI change past this phase's scope. Rather than force it in-session, the worker stopped and logged **BUG-177** with full root cause and fix shape; `BoxVertexColors.glb` renders (flat gray) instead of erroring, which is the phase's actual, bounded win.

**Held-out gate (the phase's real point):** `abandoned_warehouse_-_interior_scene.glb` — Peter's own real-world asset (`extensionsRequired: ["KHR_materials_pbrSpecularGlossiness"]`, confirmed by JSON inspection) — imports (35 materials, 26 textures wired) and renders a fully textured, non-black warehouse interior. This is the asset that motivated the whole phase; D2 is the fix that makes it work.

**P4 (D3 per-map-family samplers + D6 instancing, BUG-164/168):** The single shared, hardcoded-REPEAT `material_sampler` (`render_scene.rs:817-824`, landed `85b5bb9d` specifically to fix the striped-helmet smear) is replaced by five samplers per draw — base color, normal, metallic-roughness, occlusion, emissive — each built from the glTF texture's own `wrapS`/`wrapT` + filter, cached by a packed `MapSamplerDesc` descriptor. Default (no explicit glTF sampler) stays REPEAT+linear, byte-identical to the old hardcoded behavior — this is what keeps the helmet fix intact. `TextureSettingsTest.glb` (10 materials/objects) now shows the correct per-row divergence: clamp rows show edge-smear, repeat/mirror rows tile the checkmark correctly. D6: `EXT_mesh_gpu_instancing` is parsed via raw-JSON sniff (confirmed absent from typed support in gltf/gltf-json 1.4.1's own source) and expanded into N wired copies at summary time, so the existing 1:1 object pipeline and `OBJECT_SAFETY_MAX` apply unchanged with no new render-path code — `SimpleInstancing.glb` expands to 125 distinct scaled/rotated cube instances (4500 vertices = 125 × 36), visually confirmed as 125 separate cubes, not one mesh.

**Golden regression, diagnosed not assumed (the phase's stated real risk):** 18 of 65 prior `expect_pass` goldens shifted under the sampler change (mean_abs 0.0000–0.5552, all far under the 2.0 tolerance). Root-caused two ways, both confirmed rather than waved through: (1) assets with an explicit glTF sampler (BoxTextured, Duck, NormalTangentTest, TextureTransformTest, …) now honor their real `minFilter`/`wrapS` instead of the old hardcoded LINEAR/REPEAT — this is the intended D3 effect, verified via decode-diff and visual side-by-side (TextureTransformTest's grid texture visibly loses incorrect Repeat tiling once its real ClampToEdge sampler is honored); (2) assets with a byte-identical sampler descriptor before/after still showed tiny deterministic diffs, ruled out as runtime nondeterminism (two back-to-back renders of the same binary were bit-identical) and attributed to shader-recompile-triggered floating-point reordering, visually indistinguishable. No golden was silently re-baked without this diagnosis.

## Deviations from brief

- **Escalation, not a shortcut:** P3's D4 VERIFY-AT-IMPL found `flatten_primitive` does NOT read COLOR_0, and wiring it would require a `MeshVertex` GPU-ABI change (shared by ~30 primitives). The doc said this was "in-scope for the same phase" if missing, but the actual blast radius crosses DESIGN_DOC_STANDARD §4's escalate line (changing a public API shape the doc doesn't specify). Stopped and logged BUG-177 rather than building it in-session — judged the correct call given the standard's own rule, not a deviation from intent.
- **Bug-ID collision** (described above) — an artifact of two concurrent workstreams both minting new BUG-NNN IDs against a stale count; fixed by renumbering before landing, not discovered until the orchestrating session read the merged backlog.
- **Sampler `min_filter` mip component collapsed:** P4's `GltfFilterMode` only carries the 2-state Nearest/Linear that `manifold_gpu::GpuFilterMode` supports — no per-map mip-mode plumbing; `mip_filter` stays fixed Linear everywhere (matching pre-D3 behavior). Confirmed `TextureSettingsTest.glb` only exercises wrap, not filter, so this doesn't affect the target fixture's proof. Documented in `GltfFilterMode`'s doc comment, not silently dropped.
