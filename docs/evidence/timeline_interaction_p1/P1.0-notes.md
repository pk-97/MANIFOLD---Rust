## S1 — shift-click / cmd-click selection chrome

**Reproduced via** `selectionclips` scene (`fixtures.rs::selection_clips_scene`,
4 contiguous clips `clip-a..clip-d` at beats 0/8/16/24, each 8 beats, on layer
`select-clips`) + two new interact verbs (`click_clip`, `shift_click_clip`,
`cmd_click_clip` — see "harness extension" below).

**Shift-click range (`click_clip:clip-a;shift_click_clip:clip-d`):** console
evidence — `shift_click_clip -> extended region to 'clip-d' end=Beats(32.0)
layer_idx=0 (region.is_active=true, selected_clip_ids still =[])`. This is
sharper than the spec's own audit guessed: shift-click doesn't just leave the
anchor clip highlighted while the rest aren't — `UIState::select_region_to
_with_project` (`ui_bridge/mod.rs:519-588`) calls `UIState::set_region`
(`ui_state.rs:239-272`), which does `self.selected_clip_ids.clear()` at
line 249 unconditionally. So after a shift-click range, **zero** individual
clips are marked selected — only the region-overlay band renders (the light
inner border + per-clip recolor in `app_render.rs`'s clip-body pass reads
`self.selection.is_selected(&cr.clip_id)`, which is now false for everything).
**CONFIRMED**, sharper than hypothesized: not "one clip in the range not
highlighted" but "no clip in the range individually highlighted" — the band
is the only visible selection chrome. Root: `editing.rs:43-51`'s shift-click
arm calls the region-producing helper instead of extending the clip-id set
(exactly D2's diagnosis). PNGs: `s1_shift_range_base.png` / `_after.png`.

**Cmd-click multi-select + region-together (`click_clip:clip-a;
cmd_click_clip:clip-b;cmd_click_clip:clip-c`):** console evidence —
`selected_clip_ids=[ClipId("clip-b"), ClipId("clip-c"), ClipId("clip-a")]
region.is_active=true`. **CONFIRMED**: both authorities are live
simultaneously by design of today's code — `editing.rs:52-57`'s cmd-click arm
toggles the clip-id set AND calls `update_region_from_clip_selection_inline`
(`ui_bridge/mod.rs:481-516`), which sets `region.is_active=true` from the
selected clips' bounds whenever 2+ are selected. Both the per-clip highlight
AND the region overlay band render at once — this is what the spec's audit
called "the visual mess (hypothesis; pinned in P1.0)". Pinned: it's not a
race or a bug in the sync path, it's the literal state both call sites
produce on purpose, just never reconciled into one chrome. PNGs:
`s1_cmd_multi_base.png` / `_after.png`.

## S2 — clip renders over the header column when dragged toward beat 0

**Anchors drifted** from the spec's audit (2026-07-04 same-day audit, but
automation-lanes P4 landed additional lines since): `handle_move_drag` is now
at `interaction_overlay.rs:1551-1646` (was cited 679-775); `finalize_move_snap`
is now at `interaction_overlay.rs:1917-1956` (was cited 1045-1078).

**Model-side clamp is airtight — ruled out as the mechanism.** Per-frame drag
(`handle_move_drag`): the group beat-delta is clamped at line 1636
(`beat_delta = beat_delta.max(-self.drag_selection_min_start_beat)`), then
each dragged clip is independently clamped again at line 1641
(`(snapshot.start_beat + beat_delta).max(Beats::ZERO)`). At release,
`finalize_move_snap` applies the grid-snap delta and clamps a THIRD time at
line 1950 (`(clip.start_beat + snap_delta).max(Beats::ZERO)`). `start_beat`
provably cannot go negative at any point in this path — confirmed by reading,
not assumed.

**Both render passes that consume `visible_clip_rects` are scissored to the
tracks rect — also ruled out.** Clip bodies: `app_render.rs:3899-3902` wraps
`clip_draw::emit_clips` in `push_immediate_clip(tracks.x, tracks.y, tracks.
width, tracks.height)` / `pop_immediate_clip()`. Clip name labels: initially
suspected as the leak (called at `app_render.rs:4173`, *after* the region/
cursor/marker scissor block closes at `:4168`, with no visible bracket at the
call site) — but `clip_draw::emit_clip_names` (`manifold-renderer/src/
clip_draw.rs:217-251`) turns out to self-scissor: `push_immediate_clip(tracks.
x, ...)` at line 219 wraps the whole function, PLUS a second per-label
sub-clip to the clip's own interior at line 236. So today's code properly
bounds both passes to the tracks rect; the D7 "caller scissors, opt-in"
audit note (now at `viewport.rs:634-639`'s `timeline_overlays` doc comment,
was cited `:608`) describes the REGION/CURSOR/MARKER overlay contract, which
is ALSO correctly bracketed at `app_render.rs:4153-4168` — not unscissored.

**Hypothesis revised, not confirmed.** With the clamp math and both known
draw passes cleared, the remaining candidates are: (a) a render path I haven't
located — no evidence found of a separate "drag ghost" overlay distinct from
the live-mutated model (`handle_move_drag` mutates the real clip data directly
via `host.set_clip_start_beat`, there's no parallel unclamped preview struct);
or (b) a **one-frame lag between the model clamp and viewport scroll state**
during auto-scroll: `host.auto_scroll_for_drag(screen_pos)` (line 1566) can
adjust `scroll_x_beats` independently of the beat-delta clamp computed the
same frame, so `visible_clip_rects`' `beat_to_pixel` could transiently map a
correctly-clamped `start_beat` through a stale/adjusting scroll offset and
produce a screen x briefly left of `tracks_rect.x`, painting whatever's drawn
there over the header for that frame before the scissor state settles.
Testing (b) needs a live drag with the real `TimelineEditingHost` (the trait
has ~40 methods, `timeline_editing_host.rs:62-391` — too large a mock to
build in this phase) or an in-app repro. **Escalate to P1.4** (which owns the
S2 fix under D7's structural single-choke-point scissor): re-derive fresh
rather than trust this narrowing, and get Peter's exact repro steps
(`assume-latest-build-run`) since a code-only search did not find the leak.

## S3 — Cmd+D on 4 contiguous clips leaves a gap

**Reproduced via** `selectionclips` scene + `click_clip:clip-a;
shift_click_clip:clip-d;cmd_d` (new `cmd_d` verb — mirrors `input_host.rs:
572-628`'s `duplicate_clips` exactly: reads `selection.get_region()`,
converts via `ui_translate::selection_region_to_core`, calls
`EditingService::duplicate_clips`, executes the commands, re-selects the new
ids). Console evidence: `cmd_d -> mode=REGION (4 commands) before_ids=5
new_ids=[...]`.

**CONFIRMED the wrong-mode mechanism D3 names**: selecting the 4 clips via
shift-click (per S1) leaves `region.is_active=true` (region = [0,32) beats
spanning all 4 clips), so Cmd+D takes `EditingService::duplicate_clips`'s
REGION branch (`service.rs:688-718`, offset = `region.duration_beats()`)
instead of the individual-clip-span branch (`service.rs:719-751`, offset =
`max_end - min_beat` of the selected clips). **Caveat, precisely measured, not
guessed:** in THIS fixture the two offsets are numerically identical (region
spans exactly [0,32), the 4 clips also span exactly [0,32) with no gaps or
insert-cursor history to skew the anchor), so the copies land flush here —
the fixture doesn't reproduce a visible *gap*, only the wrong-mode branch
selection. D3's own text already anticipates this: "the gap came from the
wrong mode firing, not the wrong rounding" — a real gap needs the region's
`start_beat` to diverge from the selection's true `min_beat` (e.g. an
insert-cursor anchor left over from a prior click, or a shift-click anchored
from a non-clip-boundary position), which this static fixture doesn't carry.
The MODE bug itself is directly observed and file:line-anchored regardless.
PNGs: `s3_cmd_d_base.png` / `_after.png`.

## S5 — multi-clip move drops selection on overlap-split (PRIORITY finding)

**CONFIRMED by a throwaway test** (`manifold-editing/src/commands/clip.rs`,
reverted before commit — see gate below) exercising exactly the audit's
named suspect: `EditingService::enforce_non_overlap` (`service.rs:206-306`)
Case 4 (clip lands in the middle of an existing occupant → trim + tail split)
and `AddClipCommand::execute` (`clip.rs:279-286`, unchanged from the spec's
cited range).

**Repro:** layer with `c1` [0,10) and `c2` [60,70) as the two SELECTED/dragged
clips, `c3` [20,50) as a stationary UNselected occupant. Live-drag already
wrote the post-drag positions into the model (matches D5: preview mutates
model directly, not a separate ghost) — `c1` lands at [25,35) (inside `c3`'s
span → Case 4 split), `c2` lands at [40,50) (inside where `c3`'s SPLIT TAIL
will land, [35,50)). Processing `c1`'s overlap enforcement first (mirroring
`interaction_overlay.rs:1483-1485`'s per-snapshot loop order) returns 2
commands: `TrimClipCommand(c3: [20,50)->[20,25))` and
`AddClipCommand(tail: [35,50), a FRESH id via `clone_with_new_id()` at
`service.rs:282`)`. Executing the tail's `AddClipCommand` calls `Layer::
add_clip` (`layer.rs:522-529`), which calls `self.enforce_non_overlap_for
(&clip_id, &HashSet::new(), spb)` at **line 526 — a HARDCODED EMPTY ignore
set**, unlike the caller-supplied `ignore_ids={c1,c2}` that `EditingService::
enforce_non_overlap` respects. This second, unguarded overlap pass scans the
WHOLE layer (including `c2`, still sitting at its just-moved [40,50)) with no
protection, finds `c2` fully covered by the tail [35,50) → Case 1 → **`c2` is
deleted outright**. When the outer per-snapshot loop then reaches `c2`'s own
`enforce_non_overlap` call, the lookup fails silently (clip already gone) —
observed console line: `enforce_non_overlap(ClipId("c2")) — clip already
gone, no-op`. Final state: `ids_after = [c1, c3, <fresh-tail-id>]` — `c2` is
gone, `c2_survived = false`.

**Mechanism CONFIRMED, root is more precise than the audit guessed.** It
isn't that the overlap-split path recreates the MOVED clips' own ids (moved
clips keep their ids throughout) — it's that **`Layer::add_clip`'s internal
overlap enforcement (`layer.rs:526`) always runs with an empty ignore set,
so when it's invoked indirectly (via an `AddClipCommand` built by
`EditingService::enforce_non_overlap`'s Case-4 tail) it has no way to know
which clips are currently "protected" members of the same drag/duplicate
batch, and can silently delete/trim/split one of them if the tail's computed
geometry happens to land on it.** This is a second, INDEPENDENT overlap pass
with no ignore-list plumbing, layered underneath the first (ignore-list-aware)
one. D6's fix (id-mapping report + central selection remap) addresses the
*selection* symptom, but the more fundamental gap is that `Layer::add_clip` /
`enforce_non_overlap_for` needs an ignore-set parameter threaded through
(today: `add_clip(clip, spb)` — no ignore param at all, `layer.rs:522`) so a
tail born from one overlap resolution can't blindly re-trigger a second,
unguarded one against the very clips the first pass was told to protect.
Flagging this precisely for P1.3/P1.6 (whichever phase actually threads the
ignore-set fix) even though this phase fixes nothing.

## Harness extension

Extended `crates/manifold-app/src/ui_snapshot/interact.rs`:
- Generic `;`-chaining in `apply()` — the CLI only threads one `--interact`
  string per process, so a multi-gesture "before" scene (click, then
  shift-click) needs to happen inside one string. Splits on `;`, applies each
  part in sequence, joins descriptions with ` | `.
- `click_clip:<clip_id>:<layer_id>` — calls `UIState::select_clip` directly
  (same call `editing.rs:60`'s plain-click arm makes).
- `shift_click_clip:<clip_id>:<layer_id>` — calls `ui_bridge::
  select_region_to_with_project` (the exact function `editing.rs:46` calls),
  looking up the clip's own end-beat/layer-index first (mirrors
  `editing.rs:32-41`'s lookup).
- `cmd_click_clip:<clip_id>:<layer_id>` — calls `UIState::
  toggle_clip_selection` then `ui_bridge::update_region_from_clip_selection_
  inline` (the exact pair `editing.rs:55-57` calls).
- `cmd_d` (no args) — mirrors `input_host.rs:572-628`'s `duplicate_clips`:
  reads the current selection/region, calls `EditingService::duplicate_clips`,
  executes the returned commands on `data.project`, re-selects the new ids.

All four call the SAME production functions the real gesture dispatch calls
(`ui_bridge::editing.rs`'s arms, `input_host.rs`'s duplicate path) rather than
re-deriving the logic — same "drive the data field / call the real function,
the level under test is render reaction" convention the `automation_*` verbs
already established. New fixture: `fixtures.rs::selection_clips_scene`
(`"selectionclips"`) — one layer with 4 contiguous 8-beat clips
(`clip-a..clip-d`, ids set explicitly since `TimelineClip::new_video`'s first
arg is the display name, not the `ClipId`) plus a second layer with one clip,
so the scene still reads as a real multi-track session.
