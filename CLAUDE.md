## Hard rules

- **Shell — no `cd` prefix.** The working directory is already the project root. A `cd ... &&` prefix bypasses the Bash allowlist and prompts on every call. Different cargo target → `--manifest-path`; different cwd → a dedicated Bash call.
- **Shell — the `preToolUseBash.py` hook governs prompts; read it, don't re-derive.** [.claude/hooks/preToolUseBash.py](.claude/hooks/preToolUseBash.py) auto-allows any compound (pipes, `;`-chains, `for`/`while`, `$()`) where *every* command-position is pre-approved: read-only tools (rg/fd/jq/sort/git-read/cargo-read) or normal git/cargo *workflow* writes (add/commit/push/pull/checkout/rm/mv/…). Still prompts: destructive git (`reset --hard`, `clean`, `rebase`, `gc`), a real write inside a chain, a `$()` that runs a write, redirects to a repo path (only `/tmp/*` and `/dev/null` auto-allow), and `cd &&` prefixes. Quoting style and `|` inside a quoted regex no longer matter. **Shared-checkout guard:** a branch-switch command (`checkout`/`switch`/`merge`, not `checkout -- <paths>`) that targets the MAIN checkout — bare `git`, or `git -C <path>` resolving inside the main tree and outside `.claude/worktrees/` — auto-allows with a WARNING attached (naming the other live session) whenever another session's daemon pidfile (`.claude/daemon/verdicts/*.pid`) is live. It never asks (downgraded 2026-07-04 so orchestrations don't pause) — when you see the warning, prefer a worktree and re-read branch state from command output. Worktree-targeted commands and solo sessions are unaffected. **Landing-protocol guard:** in the main checkout, `git branch -f main ...` and any force-push targeting `main` always ask (unconditionally — these drop commits under the merge-trunk model, not just under concurrency); a plain (non-force) push or merge that lands on main gets an allow with a reminder of the fetch/merge/gate/push loop attached. Spec: `.claude/GIT_TREE_DISCIPLINE.md`. Full rationale + the echo/tail/head caveat: the `feedback_no_shell_pipes` and `feedback_no_echo_tail_head_in_bash` memories.
- **Commit-message gotcha:** backticks/`$()` in a `-m "..."` message are real bash substitution — single-quote the message, escape, or use a `<<'EOF'` heredoc, or the commit misbehaves and prompts.
- **No bare `#[allow(dead_code)]`.** Every suppression names its un-suppression trigger in a comment (design doc / phase / wiring task) or the code gets deleted. A hook enforces this at edit time.
- **All GPU via `manifold-gpu`.** Cross-platform is a product requirement, not a someday-maybe: the shipping target is native Metal on macOS and native Vulkan on Windows/Linux, each hand-written against its own API. Today `manifold-gpu` is Metal-only because `docs/VULKAN_BACKEND_DESIGN.md` (APPROVED, Phase 0 scaffold shipped in `0c5dde17`) hasn't been executed past scaffolding — that's a build-order gap, not a scope decision. Don't describe the app as "Metal-only" or "macOS-only" as if that were the design; say "Metal today, Vulkan approved-not-yet-built."
- **No new shared state.** Don't introduce `Arc<Mutex<>>` / `Arc<RwLock<>>` without approval. The content thread owns the `Project`; UI gets `Arc<Project>` snapshots.
- **All mutations through `EditingService`.** UI sends `ContentCommand::Execute(Box<dyn Command>)` or `MutateProject(Box<dyn FnOnce(&mut Project)>)`. No direct model writes from UI.
- **Generators / effects work → read `docs/DECOMPOSING_GENERATORS.md` first. Always.** The guide encodes the workflow plus every bug class that has actually bitten across the migrations to date — primitive vocabulary, mandatory GPU parity tests, coordinate conventions, what counts as "done." Working from existing primitive code as a template is not a substitute. Read the whole thing, not just §3. Skipping it means rediscovering each lesson the expensive way.
- **Decomposition: complete the §2.5 audit before proposing any new primitive.** Three required steps, all read-only: (a) survey existing primitives via `rg 'purpose: "' crates/manifold-renderer/src/node_graph/primitives/ -g "*.rs"`; (b) identify the nearest reference preset from [docs/NODE_CATALOG.md](docs/NODE_CATALOG.md) §5 / §6.1 and read it end-to-end (open the JSON, follow every wire); (c) reconcile the planned decomposition against what already exists. State the audit's findings before proposing any primitive — "this exists / this is one wire away from existing / this is genuinely new." Skipping the audit produces the "argue from snippets" anti-pattern: proposing primitives that already ship under a different name, or duplicating a pattern an existing preset already proves. Read-only audits stay in the main context — no agents (per `feedback_no_agent_for_readonly_audit`).
- **No fused single-effect or single-generator monolith nodes.** A primitive does one composable thing — a single GPU dispatch, a single DNN inference, a single FFI call, a single CPU operation. Bundling multiple distinct dispatches into a "this is the whole effect" or "this is the whole generator" kernel is the recurring migration-shortcut anti-pattern, and it's not permitted. DNN / FFI / CPU work stays at primitive granularity (depth estimate, blob detect, optical flow, envelope follower, CoreText raster) and composes into effect graphs alongside GPU atoms. The "fuse for parity" shortcut produces bundled kernels that wear primitive clothing and hide useful atoms inside; decompose instead — see `docs/DECOMPOSING_GENERATORS.md` for the bundle-vs-atom criterion.
- **Every barrier-free per-element GPU atom ships on the freeze codegen path (fusable) — no plain-WGSL fusion-boundary atoms.** Peter's standing rule (2026-07-11): all nodes, new and existing, must work perfectly with the graph compiler in full. Author a `wgsl_body` fragment + `fusion_kind`/`input_access` in the `primitive!`, build the pipeline from `standalone_for_spec::<Self>()`, and prove a generated-vs-hand parity test — never `create_compute_pipeline(include_str!(…))` as the runtime kernel (that's a fusion boundary: N dispatches where a fused run costs ~1 → dropped frames on heavy meshes at the rig). Scope test (authoritative version in `docs/ADDING_PRIMITIVES.md` §"The codegen path is mandatory"): the kernel is a barrier-free pure per-element function — one thread per output element, no workgroup memory/barriers/atomics, no cross-frame state, no CPU IO. Exempt: barriered reductions (`peak`/`luminance`) and multi-pass scan/reduce (`spawn_from_mesh`/`scatter_on_mesh`) — one kernel would break the no-monolith rule; state atoms (`temporal`); IO/bridges; `render_*` draw calls. Passes-the-test-but-codegen-can't-express-it = BLOCKED, a tracked compiler gap, never a de-facto exemption — historical case: `draw_*`'s array-into-texture read, fixed by `InputAccess::BufferIndex` (BUG-114, FUSION_SOTA_DESIGN.md D3/P4a+P4b). Existing plain-WGSL atoms are tech debt to convert.
- **Fix at the root, not the symptom — default to the fundamental fix.** When a bug has a structural cause, the fix is the structural one. Don't present the minimal patch as the recommendation and the real fix as a scary optional extra — that framing reads as laziness and Peter will call it. State the root cause, then propose the fix that removes the whole class. The minimal patch is only worth mentioning as an explicit stopgap when the root fix genuinely can't ship this session, and even then say so plainly. Inventory existing infra first (per `dont-cascade-redesign`) so "fundamental" means *correctly scoped*, not *maximally large* — but once scoped, do the real thing. See `feedback_fix_at_the_root_not_the_symptom`.
- **Commit and push when work is clean.** Don't ask permission — the user gave it durably.
- **Bug found but not fixed this session → log it in `docs/BUG_BACKLOG.md` before the session ends.** The tracker only works if ordinary sessions feed it, not just dedicated hunts (it sat untouched between its two campaigns). Symptom, root cause (or "unknown" + suspects), fix shape, next free `BUG-NNN` id.
- **Shipping = sweeping the references (supersession sweep).** A landing that completes, ships, or supersedes a design phase / bug fix / named plan must, in the same session: update the design doc's status header, update the `**Status:` line in `docs/BUG_BACKLOG.md`, then `rg` the design/plan name AND its stage labels (e.g. "Stage C", "B++") across `docs/` and the memory directory, and fix or tombstone every hit that still asserts the old state. Supersession under a *different name* is the known killer: PRESET_INSTANCE_COLLAPSE shipped 2026-06-07 and absorbed BINDING_UNIFICATION's "B+/B++/C remain", but the memories kept the old name — a month later two Fable sessions independently proposed re-designing already-shipped work from that rot. Status lives in ONE place per fact (design doc header / backlog Status line / the generated boards); memory index lines are pointers, not status.
- **Shared checkout — commit with a pathspec, never via the index.** In the main checkout use `git commit -m '…' -- <paths>`; never a bare `git commit`, and never a commit without its own pathspec. The index is shared across live sessions, so an unpathspec'd commit sweeps whatever another session has staged (bit 2026-07-04: 705 lines of another session's staged gltf work landed inside a docs commit and briefly reached main). **One exception, NEW files only:** a pathspec commit can't include untracked files — `git add -- <exactly the new paths>` first, then the pathspec commit as usual; the commit's pathspec still fences out foreign staged work. Never `add -A`, never `add .` (four sessions wedged on this in one day, 2026-07-07 — mechanics: `.claude/GIT_TREE_DISCIPLINE.md` §3b). Worktrees have their own index; this rule is about the main tree.
- **Git — `main` is the merge-based trunk; "last-known-good" is a gate property, not a fast-forward property.** (Superseded 2026-07-04: the ff-only rule above assumed one integrator at a time; concurrent orchestrator sessions each landing at different times made a clean fast-forward mathematically impossible, so every session improvised its own landing and produced twin commits — the same content merged onto main once and onto a "feat" branch again under different SHAs. See `.claude/GIT_TREE_DISCIPLINE.md` §2 and the `git-landing-protocol` memory for the incident.) Work happens on `wave/`/`lane/`/`feat/` branches. To land: fetch, merge current `origin/main` into your branch, rerun the gate (clippy + focused tests; full workspace sweep when blast radius says so), `git merge --no-ff` into main, push — if the push is rejected because someone landed first, repeat. Landings BATCH per 2–3 phases per workstream (commits stay per-phase; `.claude/GIT_TREE_DISCIPLINE.md` §2c). Two rules prevent the twin-commit failure specifically: **never cherry-pick or re-commit content that already exists as commits on a live branch** (merge the branch so SHAs stay shared — the one exception is landing the final content of a branch that is being retired immediately after, per the P4 cleanup precedent); **never delete a branch until `git merge-base --is-ancestor <tip> origin/main` confirms its commits are on main.** `git branch -f main <tip>` and force-pushes to main are anti-patterns now (they drop commits under the merge model) — the hook asks before either.
- **Agent worktrees: the slot ring is the ONLY source — hook-enforced.** Worktrees come from `python3 scripts/agent-worktree.py acquire <task-label> <branch> [--tip REF]` (one per workstream, never per phase): a fixed ring of at most 10 anonymous slots (`slot-0`…`slot-9`) that reuses the warmest idle slot, wipes a slot's `target/` past 25 GB, copies only GITIGNORED fixtures, and prints the step-0 line. It cannot grow past the cap — `POOL FULL` is a deliberate loud stop: surface it to Peter, never work around it. Raw `git worktree add` is DENIED by the Bash hook, and the Agent tool's `isolation: "worktree"` is denied by its own hook (it also bases off the default branch, not your tip) — both since the 2026-07-15 incident (19 per-task worktrees × warm targets = 455 GB, disk full). Release the slot (`release <slot-name>` as printed by acquire) at session end; a forgotten lease self-expires in 8 h. Open every worktree brief with the base-verification guard (`git log --oneline -1` must match the intended tip). The worktree-guard hook denies agent edits in the main checkout (docs included) — all edits happen in worktrees; sole exception: files git lists as unmerged during an in-progress merge in main (landing-protocol conflict resolution). Spec: `.claude/GIT_TREE_DISCIPLINE.md` (§2c = build-speed rules); hazards: the `agent-execution-playbook` memory.

## Two-thread model

- **Content thread** owns `PlaybackEngine`, `EditingService`, `ContentPipeline`, and the `Project`. Runs at project FPS (default 60).
- **UI thread** (winit) renders, handles input, presents GPU output.
- UI → Content: `ContentCommand`. Content → UI: `ContentState` snapshots. Both crossbeam **unbounded** (`app.rs`'s spawn block), consumer drains to latest — backpressure comes from drain-to-latest, not channel capacity.
- GPU output: IOSurface zero-copy triple-buffer with atomic `front_index`.

## Invariants

- Primary time model is **beats**. `Seconds` only for `in_point`, player time, delta_time, OSC, export. Function signatures take `Beats` / `Seconds` / `Bpm` newtypes — never raw `f32`/`f64`.
- `sync_clips_to_time()` is the sole authority for playback state.
- `EditingService` is the sole mutation gateway. Mutations route through `UndoRedoManager` → `Command`. Undo stack capped at 200.
- Overlap is a write-time invariant on `Layer` (`enforce_non_overlap()`).
- Phantom clips: created on NoteOn, committed on NoteOff. 5ms time guard, same-channel filter.

## Hot-path discipline

No per-frame allocations on hot paths (engine tick, sync, rendering). Use pre-allocated scratch buffers, `AHashMap` for ID lookups, and dirty-checking via `DataVersion`. GPU-side constraints (uniform alignment, texture filterability, workgroup sizes) live in `docs/MANIFOLD_GPU_ARCHITECTURE.md` — read it before touching shaders or uniforms.

## Choosing your next move — oracle discipline

Every question in a session has several possible oracles: guess, read, search, compute, run, ask git, ask the user. Pick the cheapest one that is *reliable for that question's class* — and notice that familiar is not the same as reliable. Reading and grepping always return something, which is exactly why they get overused.

- Text question ("where does this string appear") → `rg`. Meaning question ("who calls this, what implements this") → LSP. Tell: if renaming the symbol would break your search, wrong oracle.
- Behavior question ("what actually happens, in what order") → run it with printlns and read the logs. If you can observe instead of deduce, observe.
- History question ("why is it like this, when did it break") → `git log -S`, blame, read the diff that introduced it. Git is a debugger, not just a save button.
- Visual question ("does it look right") → headless render to PNG and look at it. A green test is not a look.
- Computable question (counts, offsets, alignment, "how many presets use X") → write the three-line script. Never eyeball arithmetic a machine can do exactly.
- Mechanism question (behavior governed by a hook, registry, config, codegen) → read the mechanism itself, never infer it from examples of its output.
- Negative claim ("there is no X") → run the search that would find X before asserting it. Absence is a claim like any other, and the most commonly botched one.

Above tool choice: verify one level closer to the stage than where you changed things — compiles ≠ correct ≠ looks right in the show. Scale verification with the cost of being wrong, not the size of the diff; one line in `sync_clips_to_time` outranks 200 lines of dev tooling. And "I don't know" is half an answer — the other half is naming the oracle that would resolve it.

## Tooling

- Search with `rg` not `grep`, `fd` not `find`, `ast-grep` for code-shape queries (signatures, impl blocks, macro invocations). For symbol-level questions on Rust code — "where is this defined", "what calls this", "what implements this trait" — prefer the LSP tool (`goToDefinition`, `findReferences`, `incomingCalls`, `goToImplementation`) over `rg`; it catches trait dispatch and qualified paths that text search misses.
- Runtime bugs (callbacks, event ordering, timing): add `println!`/`eprintln!`, reproduce, read logs. Static analysis is for compile errors only.
- Testing scope — the default sweep is `cargo nextest run --workspace` (the canonical runner: shared thread pool across binaries, slow-test detection; config in `.config/nextest.toml`. Plain `cargo test --workspace` works but parallelizes worse) and is **GPU-free**: 2,993 tests in ~8s warm (2026-07-11), deterministic, safe to run freely. Its meta-tests enforce repo discipline too — adding/renaming a doc requires `python3 scripts/gen_docs_index.py` or the docs-index freshness test goes red. Every GPU-readback test — the ~60 per-primitive `gpu_tests` (value-level parity), the freeze/fusion proofs, and the `gpu_proofs` integration binary (alpha-contract sweep + generator smoke) — lives behind the `gpu-proofs` cargo feature and is OFF by default. `crate::test_device()` itself only compiles under that feature, so the GPU tests aren't even built in the default sweep. Run them **deliberately** when you touch GPU-tested code: `cargo test -p manifold-renderer --features gpu-proofs` for the whole GPU suite, `... --features gpu-proofs <module_path>::gpu_tests` to narrow to one primitive, or `... --test gpu_proofs --features gpu-proofs` for the alpha+smoke binary. What warrants a GPU run: a primitive's shader/kernel, the graph runtime, `manifold-gpu`, the freeze compiler, shared WGSL headers, or a completed decomposition. Those runs are minutes-long, stay on `cargo test` — never nextest, whose process-per-test model defeats the in-process `test_device` lock that serializes device access — and can flake under device contention (a hard GPU-error check now lives in `commit_and_wait_completed`); the default sweep and focused `--lib` runs are seconds. When unsure whether a change touches the GPU path, run the feature build too — the cost of running it unnecessarily is far less than shipping a GPU regression that the default sweep can no longer catch.
- Linting: always clippy before commit. In a worktree, scope it — `cargo clippy -p <touched crates> -- -D warnings` (a `--workspace` clippy in a cold worktree is a second full build); the full `cargo clippy --workspace -- -D warnings` runs at landing time in the warm main checkout, where it IS cheap (~10–18s measured 2026-07-11). The landing sweep also runs `cargo deny check bans` (<1s) — `deny.toml` is the actual enforcement of the wgpu/metal dependency bans. Lint severity is code-versioned in `[workspace.lints]` (root Cargo.toml) + `clippy.toml`, not command-line convention.
- Authoring graph JSON (effect/generator presets, imports): pre-flight with `graph_tool validate <file.json> --kind effect|generator` and `graph_tool fusion <file.json>` (`crates/manifold-renderer/src/bin/graph_tool.rs`) before wiring it into the app — see `docs/GRAPH_TOOLING_DESIGN.md`.
- Inspecting or modifying `.manifold` **project files** (tempo map, audio clips, raw project.json): `project_tool` (`crates/manifold-io/src/bin/project_tool.rs`) — mutations are surgical raw-JSON edits validated through the real loader and written with the app's archive writer (a registry-less typed round-trip DROPS params; never hand-edit the ZIP). `tempo at` is the beat→seconds verification oracle for imported tempo maps.
